## 1. 背景と目的

### 1.1 背景（現状課題）
- プロジェクトごとにテスト管理フォーマット・運用方法がバラバラ
- 必要な情報（更新日時、更新者など）が不足している
- 「いつ書いた情報か」「最新版かどうか」が分からない
- チーム内での確認・認識合わせに時間がかかっている

### 1.2 目的
- テスト管理のフォーマットと運用を統一する
- 更新日時・更新者を必ず残し、情報の鮮度を明確にする
- 進捗状況とテスト一覧を誰でもすぐに把握できるようにする

---

## 2. 管理単位の定義

### 2.1 プロジェクト単位
- **リリース単位**で管理する
- 1リリース = 1回の出荷 / 本番反映のまとまり

### 2.2 テストケース粒度
- **画面単位**
- 1テストケース = 1画面
- 初期段階では画面単位に固定し、必要に応じて将来分割する

## 3. ユーザーと役割

| 役割 | 主な操作 |
|---|---|
| テスター | テスト実行、結果記録、証跡添付 |
| システムエンジニア | 進捗確認、テスト設計レビュー |
| プログラマー | 不具合確認、修正後の再テスト確認 |

---

## 4. ステータス定義（正式）

### 4.1 テストケースステータス

| ステータス | 定義 | 完了扱い | 集計対象 |
|---|---|---:|---:|
| 未着手 | 一度も実行していない | × | ○ |
| 実施中 | 実行途中・確認中 | × | ○ |
| 合格（Pass） | 期待結果を満たした | ○ | ○ |
| 不合格（Fail） | 期待結果を満たさない | × | ○ |
| ブロック | 環境・前提不足で実行不可 | × | ○ |
| 保留 | 仕様・判断待ち | × | ○ |
| 対象外（N/A） | 今回のリリース対象外 | ○ | × |

---

## 5. 進捗定義（方式①）

### 5.1 集計ルール
- 対象外（N/A）は **分母から除外**
- 完了 = 合格（Pass）のみ

### 5.2 計算式
- 分母（対象件数）  
  = 全テストケース数 − 対象外件数
- 分子（完了件数）  
  = 合格件数
- 進捗率（%）  
  = 合格件数 ÷ 対象件数 × 100

### 5.3 補助指標
- 未完了件数 = 対象件数 − 合格件数
- リスク件数 = 不合格 + ブロック + 保留

---

## 6. MVP機能要件

### 6.1 テストケース管理
- 画面単位でテストケースを登録・編集・一覧表示できる
- ステータス・担当者・更新日時でフィルタ可能

### 6.2 テスト実行結果管理
- Pass / Fail / Blocked / 保留を記録できる
- コメント・証跡（添付 or URL）を残せる

### 6.3 進捗可視化
- リリース単位で進捗率を自動算出
- 未完了・リスクケースを一覧表示できる

## 7. 画面要件（MVP）

### 7.1 リリース一覧画面
- リリース名
- 期間
- 進捗率
- 未完了件数
- 最終更新日時

### 7.2 テストケース一覧画面
**表示項目**
- 画面名
- ステータス
- 担当者
- 最終実行結果
- 最終実行日時
- 更新日時

**フィルタ**
- ステータス
- 担当者
- 未完了のみ
- リスクのみ（Fail / Blocked / 保留）

### 7.3 テストケース詳細画面
- 画面名
- URL / 画面パス
- 確認観点
- 期待結果
- ステータス
- 実行履歴
- 証跡
- 作成日時 / 更新日時
- 作成者 / 更新者

## 9. 運用フロー（統一ルール）

1. リリースを作成する
2. 画面単位でテストケースを登録する
3. 対象外の画面は N/A に設定する
4. テストを実行し、結果・証跡を記録する
5. Fail / Blocked は未完了として残す
6. 再テストで Pass になった時点で完了
7. 全対象ケースが Pass になったらリリース完了

---

## 10. MVP受け入れ条件
- 画面単位でテストケースを管理できる
- リリース単位で進捗率が自動計算される
- 更新日時・更新者が必ず確認できる
- 未完了・リスクケースが即座に把握できる
